# markovian-bandwidth-law
Numerical validation and experimental reproduction for the single-window quantum bandwidth law (QBL) in Markovian open systems. MQBL-SWV.py tests linear scaling with log‑bandwidth Δ, semiclassical ε², and geometric slope K. MQBL-EV.py computes the visibility separatrix from cold‑atom data (Barontini et al.) on the same spectral ruler.

This archive contains two Python programs:

## 1. MQBL-SWV.py
MQBL-SWS (Markovian QBL Single-Window Validator) generates single‑window, paper‑aligned diagnostics for geometry–memory separation and bandwidth laws using only instrument and window settings. It computes thermal and semiclassical scales, builds a rectangular‑window frequency ruler, evaluates a geometry‑locked factor from an L2‑normalized point‑spread function paired with a smooth third‑derivative profile, and, when enabled, constructs a time‑domain kernel and a bilateral frequency‑domain susceptibility to report a family of gain factors. It then assembles the main response on the selected band, prints a compact console report, and saves figures plus CSV tables for each window.

You can run the program on Linux, macOS, or Windows (64‑bit) with Python 3.8 or newer. A modern multi‑core CPU and 4 GB of RAM are sufficient for the default configuration; 8 GB is recommended when you increase spectral or time‑grid sizes. Disk usage is modest; leave several hundred megabytes free to hold figures and CSV outputs. Install the required packages NumPy, SciPy, and Matplotlib in your environment, for example by running pip install numpy scipy matplotlib.

To use the program, place the script in a working folder and run it with python your_script.py. No input files are required; all settings are taken from the configuration object created in the main block. The default configuration produces a multi‑window report and writes all outputs to a directory named figs. If you want to customize the run, edit TheoryConfig before calling the driver. The most common adjustments are the output directory, the window length that sets the frequency ruler, the instrument velocity mode (local or thermal), whether to compute the time/frequency kernel and gain factors, and whether to apply calibration so that the in‑band gain is unity on the chosen band. You can also enable or disable drift calibration, change the number of windows and the gap between them for multi‑window attenuation, and choose the point‑spread function normalization. After editing, run the script again; the flow and outputs remain the same.

Results are saved under the output directory (default figs) with one sub‑series per window index. For each window you will find a short overview figure with key scalars, a spectral‑band panel showing the ruler weight and the bilateral kernel, a time‑kernel panel with the triangle integral and its cumulative, a geometry panel with the point‑spread function, profile, and their product, a window‑filter panel with a cumulative reference, phase‑magnitude views of the kernel, a cumulative log‑integral panel for the gain channel, the Hermite–Gaussian basis used by the kernel, a bar plot comparing gain definitions, a validity‑checks panel against the configured tolerances, and a remainder‑bounds panel. Each figure has a matching CSV file with the exact numbers plotted, and a per‑window summary CSV consolidates scales, edges, gains, response, slopes, and remainder values. The program prints a concise console report and, if you request multiple windows, a final attenuation summary that combines single‑window responses into coherent and root‑mean‑square totals.

If you encounter slow runs or large memory use, reduce the kernel grids, shorten the window list, or disable the kernel path by setting compute_J to false. If you need tighter accuracy, increase the time and spectral grid sizes, then rerun and compare the per‑window summary and slope diagnostics. The default settings are suitable for headless servers; all plots are saved to disk and no interactive backend is required.

## 2. MQBL-EV.py
MQBL-EV (Markovian QBL Experimental Validator) reproduces a single‑window phase‑boundary analysis using two experimental data files from the cited cold‑atom study, then builds a portable geometric model to predict response thresholds across drive amplitude and velocity. It reads Fig2C.csv to extract a momentum‑scale proxy from many lineouts via a robust center‑of‑mass procedure, and it reads Fig3.csv to load the cloud of experimental drive amplitude and velocity points. Using these inputs together with a Gaussian point‑spread function and a Gaussian barrier model, the script estimates a geometry‑locked prefactor, constructs a single‑pole susceptibility on the common rectangular‑window ruler, computes a band‑averaged gain and the logarithmic span for each velocity, and produces a predicted threshold curve along with a continuous ratio map that shows normalized response versus threshold. Figures and CSV tables are written to figs_paper_phase.

You can run on Linux, macOS, or Windows (64‑bit) with Python 3.8 or newer. A modern multi‑core CPU and 4 GB of RAM are sufficient for the default settings; the job is light on memory and storage, though you should leave several hundred megabytes free for figures and CSVs. Required packages are NumPy, Pandas, and Matplotlib. Install them in your environment, for example with pip install numpy pandas matplotlib. The script sets Times New Roman and STIX math fonts for publication‑quality figures; if these fonts are not installed, Matplotlib will fall back automatically, so appearance may differ slightly.

Place Fig2C.csv and Fig3.csv in the working folder and run the script as python your_script.py. The program expects Fig2C.csv to contain one frame per row with 266 columns per lineout; it will transpose if the file is 266 rows by many columns and will warn if the number of frames differs from the built‑in expectation. Non‑finite values are zeroed, negatives are clipped to zero, and a mild outlier guard limits saturated frames before estimating the momentum scale. The program expects Fig3.csv with three numeric columns labeled in the script as amplitude in kelvin‑equivalent units, velocity in meters per second, and a third column that is not used by the boundary fit; two header rows are skipped. If either file is missing or the shapes cannot be reconciled with the built‑in assumptions, the script raises a clear error.

By default the program writes all outputs to figs_paper_phase. The main figure single_boundary_with_cloud.png shows the continuous ratio map, a unit‑level contour and the predicted threshold curve, overlaid with the experimental cloud; the color bar encodes the ratio of the predicted response to the chosen threshold level. The script also exports the per‑velocity band‑averaged gain and logarithmic span, the susceptibility samples on the operational ruler, the geometry integrand used to build the prefactor, and the amplitude threshold samples on the same drive‑amplitude grid. A metadata table with all key scalars and configuration parameters is saved as run_meta.csv, and the console prints the output directory together with a compact summary of the inferred geometry slope and the span of the predicted amplitude threshold. If you wish to adjust the physics model or plotting ranges, edit the parameter block in main: the window length, instrument length, edge‑width and point‑spread‑function widths, bath temperature, instrument and maximum velocities, and any caps can be changed there, as can the amplitude and velocity grid sizes and ranges.

Data source and citation. Fig2C.csv and Fig3.csv correspond to Fig. 2c and Fig. 3 in the publication: G. Barontini, V. Naniyil, J. P. Stinton, D. G. Reid, J. M. F. Gunn, H. M. Price, A. B. Deb, D. Caprioli, and V. Guarrera, Observation of Fermi Acceleration with Cold Atoms, Phys. Rev. Lett. 135, 025201 (2025), doi: 10.1103/nrjv-pwy1, and the underlying dataset: G. Barontini, J. Stinton, and V. Naniyil, Fermi acceleration with cold atoms, Rev. Mod. Phys. (2025), 10.25500/edata.bham.00001224.
